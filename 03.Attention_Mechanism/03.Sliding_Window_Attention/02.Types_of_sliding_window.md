### 🧩 1. Chunked Sliding Window Attention (Mistral Trick)

![Image](https://miro.medium.com/v2/resize%3Afit%3A1400/1%2AY4uClaGkSfrx4SlUsocg3Q.png)

![Image](https://media.licdn.com/dms/image/v2/D5612AQF7_xOtM7surQ/article-cover_image-shrink_600_2000/article-cover_image-shrink_600_2000/0/1721187600392?e=2147483647&t=-5YcxgthRd4M03OBvv3upOZMQnbib-Ob4TQee2ZXKvU&v=beta)

![Image](https://miro.medium.com/v2/resize%3Afit%3A1400/1%2AQFDYaQ_3yh_5NkZjJzGT8A.png)

![Image](https://miro.medium.com/v2/resize%3Afit%3A2000/1%2Ay0yf7AO0BEcdZEHdV9uUwQ.png)

This method was popularized by **Mistral 7B**.

Instead of computing attention **token-by-token**, the sequence is split into **blocks (chunks)** and attention is computed between **neighboring chunks**.

### Idea

Sequence

```
[0–127] [128–255] [256–383] [384–511]
```

Each chunk attends to:

```
current chunk + previous chunk
```

Example:

```
Chunk 2 → Chunk 1 + Chunk 2
Chunk 3 → Chunk 2 + Chunk 3
```

So tokens in chunk 3 can attend to **256 previous tokens**.

---

### Why this is faster

GPU kernels work best with **large contiguous blocks**.

Instead of:

```
token → window tokens
```

we compute:

```
block → block
```

This allows **FlashAttention kernels** to work efficiently.

---

### Complexity

If

```
chunk_size = C
window = 2C
```

Then attention cost becomes

$$
O(T × C)
$$

instead of

$$
O(T^2)
$$

Example

| Context | Chunk | Compute          |
| ------- | ----- | ---------------- |
| 32k     | 128   | ~4M interactions |

---

### PyTorch Idea

```python
def chunk_attention(q, k, v, chunk_size):

    B, H, T, D = q.shape

    outputs = []

    for start in range(0, T, chunk_size):

        end = start + chunk_size

        q_chunk = q[:, :, start:end]

        k_chunk = k[:, :, max(0, start-chunk_size):end]
        v_chunk = v[:, :, max(0, start-chunk_size):end]

        attn = torch.matmul(q_chunk, k_chunk.transpose(-2,-1)) / (D ** 0.5)

        attn = torch.softmax(attn, dim=-1)

        out = torch.matmul(attn, v_chunk)

        outputs.append(out)

    return torch.cat(outputs, dim=2)
```

### 🔄 2. Ring Attention

![Image](https://miro.medium.com/v2/resize%3Afit%3A1400/1%2AcfyQG7ghO7b4gb4RMgCz6w.png)

![Image](https://coconut-mode.com/KV-overlap-large.gif)

![Image](https://miro.medium.com/v2/resize%3Afit%3A1400/1%2ANwZmBmInnlSXN9Ice6oncg.png)

![Image](https://miro.medium.com/v2/resize%3Afit%3A1400/1%2AjQEDGpf5Gy2-pM0FLSDG8A.png)

Ring Attention is used for **multi-GPU training** of long-context transformers.

Instead of copying the entire sequence to every GPU, the sequence is **partitioned across GPUs**.

Example:

```
GPU0 → tokens 0-8191
GPU1 → tokens 8192-16383
GPU2 → tokens 16384-24575
GPU3 → tokens 24576-32767
```

During attention computation, **KV tensors move around GPUs in a ring**.

```
GPU0 → GPU1 → GPU2 → GPU3 → GPU0
```

Each GPU receives KV blocks from neighbors and computes partial attention.

---

### Why it works

Normally attention needs

```
Q × K^T
```

But K may be on another GPU.

Ring attention sends **K,V blocks sequentially around GPUs**.

Memory per GPU becomes

```
O(T / N)
```

instead of

```
O(T)
```

Where **N = number of GPUs**.

---

### Benefits

| Feature                 | Benefit           |
| ----------------------- | ----------------- |
| Distributed memory      | large context     |
| Communication efficient | minimal transfers |
| Scalable                | 1M+ tokens        |

### 🪟 3. Dilated Sliding Window Attention

![Image](https://klu.ai/_next/image?q=100&url=%2F_next%2Fstatic%2Fmedia%2Fklu-sliding-window-attention.4bed727d.png&w=3840)

![Image](https://miro.medium.com/v2/resize%3Afit%3A1400/1%2AIRwSKQIpHDX0ySJFPJOK6A.png)

![Image](https://isee-ai.cn/~jiaojiayu/model.jpg)

![Image](https://www.researchgate.net/publication/375843639/figure/fig4/AS%3A11431281217023676%401704990391687/Illustration-of-the-query-key-value-structure-between-Self-Attention-SA-and-Dilated.tif)

Dilated attention increases the **effective receptive field** without increasing compute.

Instead of attending to **every token in window**, it attends to **spaced tokens**.

---

### Example

Window size = 8
Dilation = 2

Instead of

```
t8 → t0 t1 t2 t3 t4 t5 t6 t7
```

It attends to

```
t8 → t0 t2 t4 t6
```

---

### Mathematical Pattern

For token **i**

$$
j = i - d × k
$$

Where

- **d** = dilation
- **k** = step index

---

### Benefit

We can cover **large context cheaply**.

Example:

| Window | Dilation | Effective Context |
| ------ | -------- | ----------------- |
| 128    | 1        | 128               |
| 128    | 4        | 512               |
| 128    | 8        | 1024              |

---

### PyTorch Mask Example

```python
def dilated_mask(seq_len, window, dilation):

    mask = torch.full((seq_len, seq_len), float("-inf"))

    for i in range(seq_len):

        for k in range(window):

            j = i - k * dilation

            if j >= 0:
                mask[i, j] = 0

    return mask
```

### 🌊 4. Streaming Sliding Window Attention

![Image](https://klu.ai/_next/image?q=100&url=%2F_next%2Fstatic%2Fmedia%2Fklu-sliding-window-attention.4bed727d.png&w=3840)

![Image](https://av-eks-lekhak.s3.amazonaws.com/media/__sized__/article_images/image_qJ4o5Fd-thumbnail_webp-600x300.webp)

![Image](https://www.researchgate.net/publication/375771237/figure/fig1/AS%3A11431281285463734%401729735037308/Transformer-architectureVaswani-etal-2017-On-the-left-side-the-encoder-component.png)

![Image](https://miro.medium.com/v2/resize%3Afit%3A1200/0%2Af4KUOX2hNfP36kR_.png)

Streaming attention is designed for **infinite sequences or live streams**.

Example tasks:

- speech recognition
- real-time chat
- video understanding

---

### Problem

Normal attention requires storing **all past tokens**.

Memory grows with time.

---

### Solution

Keep only the **recent window**.

Example

```
window = 4096
```

When new tokens arrive:

```
drop oldest tokens
append new tokens
```

---

### KV Cache Update

Instead of storing

```
KV = [0 ... t]
```

we store

```
KV = [t-4096 ... t]
```

---

### Example

Step 1

```
tokens = [0..4096]
```

Step 2

```
tokens = [1..4097]
```

Step 3

```
tokens = [2..4098]
```

This acts like a **rolling memory buffer**.

---

### PyTorch Streaming KV Cache

```python
class StreamingKVCache:

    def __init__(self, window):
        self.window = window
        self.k = None
        self.v = None

    def update(self, new_k, new_v):

        if self.k is None:
            self.k = new_k
            self.v = new_v
        else:
            self.k = torch.cat([self.k, new_k], dim=1)
            self.v = torch.cat([self.v, new_v], dim=1)

        if self.k.size(1) > self.window:
            self.k = self.k[:, -self.window:]
            self.v = self.v[:, -self.window:]

        return self.k, self.v
```

### ⚡ Comparison

| Method           | Main Goal              |
| ---------------- | ---------------------- |
| Chunked Window   | GPU efficiency         |
| Ring Attention   | multi-GPU scaling      |
| Dilated Window   | larger receptive field |
| Streaming Window | infinite sequences     |

### 🧠 Real Systems Combine Them

Modern LLMs combine multiple tricks:

Example architecture:

```
Sliding window
+ FlashAttention
+ KV cache
+ Chunked attention
```

Used in models like:

- **Mistral 7B**
- **Mixtral 8x7B**
- **Gemma**

---

💡 **Key insight**

All these techniques solve one core problem:

$$
\text{Transformer Attention} = O(T^2)
$$

These methods reduce it closer to

$$
O(T)
$$

which is why **modern LLMs can handle extremely long contexts**.
